1Handle Availability Check — Frontend

Handle checking is implemented entirely in SignUpPage.jsx, using checkHandleAvailable() from useOnboarding.js. AuthPage.jsx is a login-only screen and has no involvement in handle validation.

When it fires

Trigger Active? Detail
While typing ✓ Yes useEffect on handle state — debounced 500 ms. Fires on every keystroke after the debounce window.
On blur (field focus-out) ✗ No No onBlur handler exists on the handle input.
Before submit ✓ Yes An explicit second call to checkHandleAvailable() runs inside handleSignUp() before supabase.auth.signUp() is invoked. This guards against stale debounce state.

How the check works

checkHandleAvailable() in useOnboarding.js calls the check_handle_available Supabase RPC. The RPC is SECURITY DEFINER, so it bypasses Row Level Security and works for unauthenticated users on the signup page (a direct table query would return empty results due to RLS, creating a false "available" result).

-- check_handle_available_fn.sql
CREATE OR REPLACE FUNCTION check_handle_available(
  p_handle       TEXT,
  p_current_user_id UUID DEFAULT NULL
)
RETURNS BOOLEAN
SECURITY DEFINER
AS $$
BEGIN
  RETURN NOT EXISTS (
    SELECT 1 FROM profiles
    WHERE LOWER(handle) = LOWER(TRIM(p_handle))
    AND (p_current_user_id IS NULL OR user_id != p_current_user_id)
  );
END;
$$;
✅
Case-insensitive matching

The LOWER() wrapping means "Lal" and "lal" are treated as the same handle.

✅
Fails closed on error

Any RPC error or exception returns { available: false }, preventing a false positive from blocking a user into a taken handle.

2Server-Side Duplicate Protection

No UNIQUE constraint on profiles.handle is defined in any local SQL migration file. However, create_profile_if_missing (the post-confirmation recovery RPC) explicitly catches WHEN unique_violation and comments "Handle was taken — fall back to generated handle".

⚠️
Constraint exists but is not version-controlled

The UNIQUE constraint on handle almost certainly exists in the live Supabase schema (set up via the dashboard), but it does not appear in any supabase/migrations/ file. If the database is ever recreated from the local migrations alone, the constraint would be missing.

Race condition window

The final handle re-check and the actual account creation are not atomic. The sequence is:

User A → checkHandleAvailable("zayd") → returns true User B → checkHandleAvailable("zayd") → returns true User A → supabase.auth.signUp() → trigger inserts handle = "zayd" User B → supabase.auth.signUp() → trigger hits unique_violation ↳ If UNIQUE constraint exists: User B silently receives handle = "user_ab3f9c12" with no error shown. ↳ If no UNIQUE constraint: both users end up with handle = "zayd" — a true duplicate.
🔴
Silent handle swap is a UX problem

When a collision occurs, the user is never informed that their chosen handle was replaced. They complete signup and later discover they have a generated handle (user_ab3f9c12) instead of the one they picked. The signup screen shows no error.

3Full Signup Flow

AuthPage.jsx is a login-only screen. Signup is entirely self-contained in SignUpPage.jsx. The flow is:

SignUpPage.jsx ↓ checkHandleAvailable() — debounced while typing ↓ Submit button clicked ↓ checkHandleAvailable() — final synchronous re-check ↓ supabase.auth.signUp({ email, password, options.data: { ... } }) ↳ Supabase creates auth.users row with raw_user_meta_data ↓ DB trigger fires (handle_new_user) — inserts into profiles (see §5) ↓ Email confirmation sent to user ↓ User clicks link → CheckEmailScreen.jsx or LoginPage.jsx ↓ create_profile_if_missing() RPC called with metadata as fallback ↳ Reads desired_username, desired_handle, desired_avatar_index from user_metadata

4Metadata Storage During Signup

Confirmed. SignUpPage.jsx passes all three fields in options.data, which Supabase stores as raw_user_meta_data on the auth.users row and exposes as user.user_metadata on the client.

// SignUpPage.jsx — lines 155–166
const { data, error } = await supabase.auth.signUp({
  email: email.trim().toLowerCase(),
  password,
  options: {
    emailRedirectTo: redirectUrl,
    data: {
      desired_username:     trimmedName,
      desired_handle:       trimmedHandle,
      desired_avatar_index: avatarKeyToIndex(avatarKey),  // integer
    },
  },
});
✅
Metadata is durable

Because the values are stored in auth.users.raw_user_meta_data on the server, they survive a fresh device install. The profile recovery code in CheckEmailScreen.jsx and LoginPage.jsx can read them without relying on localStorage.

5Database Trigger Behaviour

The trigger function handle_new_user is not present in any local SQL migration file. It exists only in the live Supabase dashboard and cannot be directly inspected from this codebase. The following conclusions are drawn from production evidence.

Does the trigger read desired_* metadata?

🔴
Almost certainly not

Multiple accounts in the production leaderboard display username = "User_27bd80" and username = "User_ad1005" — placeholder values consistent with a trigger that ignores raw_user_meta_data and falls back to defaults. If the trigger correctly read desired_username, these accounts would show real names.

What the trigger appears to produce

ColumnValue WrittenSource
username 'User_' + first 6 chars of user_id Trigger default — metadata ignored
handle 'user_' + first 8 chars of user_id or NULL Trigger default — metadata ignored
avatar 0 (first avatar) Trigger default — metadata ignored
xp, coins, streak 0 Correct defaults

Recovery path

The app recovers the correct username/handle/avatar through two mechanisms:

  1. create_profile_if_missing RPC — called post-email-confirmation. Reads desired_username, desired_handle, desired_avatar_index from user_metadata and patches the profile row.
  2. useUserStore.init() recovery — detects /^User($|[_\s\d])/i pattern and overwrites with metadata values on every login.
⚠️
Recovery is not guaranteed to be instant

Between trigger creation and the first login + email confirmation, the user's profile row exists in the DB with placeholder values. Any query that runs in that window (leaderboard, friend search, friend list) will display the placeholder name and default avatar for that user.

7UI-Only Validation Risk

From what is visible in the local codebase, handle uniqueness is protected by:

  • The check_handle_available RPC (a plain SELECT, not enforced by a constraint)
  • A probable — but unconfirmed from local files — UNIQUE constraint in the live DB

If the UNIQUE constraint does not exist, two simultaneous submissions with the same handle would both succeed, creating a true duplicate. There would be no error and no notification to either user.

If the UNIQUE constraint exists, a collision triggers the unique_violation fallback in create_profile_if_missing, silently replacing the chosen handle with a generated one. The user is never told.

2. Field: streak_active

Key finding: streak_active is never stored in local Zustand state and is never read back by the client. It exists exclusively as a Supabase column written by the client and consumed server-side by the send-daily-notifications Edge Function to filter which users receive daily push notifications.

Value formula:

streakActive && challengePending  →  "AS_CP"   (active streak + challenge pending)
streakActive only                 →  "AS"      (active streak)
challengePending only             →  "NS_CP"   (no streak + challenge pending)
neither                           →  "NS"      (no streak)

How streakActive is locally calculated (pushTags.js line 61):

const today     = new Date().toDateString();
const yesterday = new Date(Date.now() - 86400000).toDateString();
const streakActive = lastStudyDate === today || lastStudyDate === yesterday;
Source of truth: streakActive for the OneSignal tag is computed entirely in memory from lastStudyDate — it does NOT read the streak_active Supabase column. The two values should agree but are calculated independently.

7. Confirmation Answers

Is streak calculated client-side only?
Yes Every streak mutation (recordDailyActivity, checkStreakOnAppOpen, repairStreak, skipRepair) runs entirely in Zustand client state first. Supabase writes follow as fire-and-forget updates. There is no server-side function that calculates or increments streak.
Is Supabase used as source of truth?
Partially Supabase stores the last known values and is the sync source when the app opens. mergeProfileData → setFromCloudSync pulls cloud values and applies them when cloud updated_at is newer than the local iq_last_cloud_sync timestamp. A _localStreakTs guard prevents cloud from overwriting a streak that was mutated more recently on-device.
Is OneSignal reading from Supabase at all?
No OneSignal tags are set purely from in-memory Zustand state via _getProgressState(). The streak_active Supabase column is consumed server-side by the send-daily-notifications Edge Function to decide who to notify — OneSignal itself never queries Supabase.
Are tags purely client-set?
Yes Every OneSignal.User.addTag(...) call originates in src/services/pushTags.js from local Zustand state. The server-side Edge Function reads streak_active from Supabase to decide who to notify, but the tag values on each device are written by that device's client only.
